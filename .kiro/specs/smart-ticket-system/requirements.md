# 智能工单系统需求文档

## 1. 项目概述

开发一个基于 LangChain 的智能客服工单系统，支持三个核心功能：订单物流查询、催单、退单。系统采用分层架构：
- **UI 层**：Python FastAPI + HTML 页面
- **用户层**：Java Spring Boot（认证、会话管理、负载均衡）
- **核心业务层**：Python FastAPI（意图识别、业务逻辑）

## 2. 用户故事

### 2.1 登录认证
作为用户，我希望能够通过账号密码登录系统，以便安全地使用智能客服功能。

### 2.2 物流查询
作为用户，我希望能够通过输入订单号查询实时物流状态，以便了解我的订单配送进度。

### 2.3 催单
作为用户，我希望能够催促订单处理，系统能够创建工单并通知客服，以便加快订单处理速度。

### 2.4 退单
作为用户，我希望能够取消订单，系统能够校验订单状态并处理退款，以便便捷地取消不需要的订单。

## 3. 功能需求

### 3.1 认证模块
- **FR1.1**: 系统必须提供登录页面，用户可输入账号密码
- **FR1.2**: 账号密码必须发送到 Java 后端进行验证
- **FR1.3**: 登录成功后系统必须返回 Token
- **FR1.4**: 账号密码必须配置在 Java 配置文件中
- **FR1.5**: 认证服务必须使用抽象接口，便于后期对接业务系统
- **FR1.6**: 支持配置模式切换（配置文件/业务系统对接）

### 3.2 意图识别模块
- **FR2.1**: 系统必须使用 LangChain + Pydantic 进行结构化输出
- **FR2.2**: 系统必须识别三种意图：logistics(物流查询)、urgent(催单)、cancel(退单)
- **FR2.3**: 系统必须提取订单号实体（格式：ORD+数字）
- **FR2.4**: 系统必须提取用户诉求细节
- **FR2.5**: 当置信度低于 0.7 时，系统必须返回澄清问题

### 3.3 物流查询模块
- **FR3.1**: 系统必须支持输入订单号进行查询
- **FR3.2**: 系统必须查询订单状态
- **FR3.3**: 对于已发货订单，系统必须查询物流轨迹
- **FR3.4**: 系统必须输出格式化物流信息，包括：最新状态、预计送达时间、最近 3 条轨迹

### 3.4 催单模块
- **FR4.1**: 系统必须支持多轮对话收集订单号和催单原因（可选）
- **FR4.2**: 系统必须创建工单（Ticket）并保存到存储
- **FR4.3**: 系统必须通知客服（打印日志模拟）
- **FR4.4**: 系统必须返回工单号、预计处理时间、客服联系方式

### 3.5 退单模块
- **FR5.1**: 系统必须支持多轮对话收集订单号和退单原因
- **FR5.2**: 系统必须校验订单状态：
  - 已取消状态：拒绝退单，提示已取消
  - 已签收状态：拒绝退单，提示走售后退货流程
  - 其他状态：允许取消
- **FR5.3**: 系统必须调用取消 API 并更新订单状态
- **FR5.4**: 系统必须触发退款流程
- **FR5.5**: 系统必须返回取消结果、退款金额、到账时间

## 4. 数据层需求

### 4.1 数据抽象层
- **DR1**: 所有业务数据（订单、物流、工单）必须通过抽象接口访问
- **DR2**: 初期使用模拟数据（Mock Data），后续可对接真实业务系统
- **DR3**: 数据接口必须定义清晰的协议，便于后续替换实现

### 4.2 模拟数据要求
- **DR4**: 订单数据需包含：订单号、订单状态、金额、创建时间
- **DR5**: 物流数据需包含：物流轨迹、状态更新、预计送达时间
- **DR6**: 工单数据需包含：工单号、优先级、状态、创建时间、关联订单

## 5. 非功能需求

### 5.1 技术约束
- **NFR1**: UI 层必须使用 Python FastAPI + HTML
- **NFR2**: 用户层必须使用 Java Spring Boot
- **NFR3**: 核心业务层必须使用 Python 3.9+
- **NFR4**: AI 框架必须使用 LangChain + OpenAI API（或兼容的 LLM）
- **NFR5**: 数据存储使用内存存储（Dict 模拟），预留数据库接口
- **NFR6**: 意图识别必须使用大模型，禁止硬编码规则

### 5.2 架构约束
- **NFR7**: UI 层通过 HTTP 调用 Java 用户层 API
- **NFR8**: Java 用户层通过 HTTP 调用 Python 核心业务层 API
- **NFR9**: Java 用户层为单体应用，预留微服务改造接口
- **NFR10**: 认证服务必须支持抽象，便于后期对接业务系统

## 6. 验收标准

### 6.1 登录认证验收标准
- [ ] 用户可访问登录页面
- [ ] 输入正确账号密码后登录成功，跳转聊天界面
- [ ] 输入错误账号密码后提示登录失败
- [ ] 登录 Token 正确返回并保存
- [ ] 未登录用户无法访问聊天界面

### 6.2 物流查询验收标准
- [ ] 用户输入有效订单号后，系统返回格式化物流信息
- [ ] 系统正确显示最新状态、预计送达时间
- [ ] 系统正确显示最近 3 条物流轨迹
- [ ] 用户输入无效订单号时，系统给出友好提示

### 6.3 催单验收标准
- [ ] 系统通过多轮对话收集完整信息
- [ ] 系统成功创建工单并返回工单号
- [ ] 系统返回预计处理时间和客服联系方式
- [ ] 客服通知日志正确打印

### 6.4 退单验收标准
- [ ] 系统正确校验各种订单状态
- [ ] 已取消订单被正确拒绝并提示
- [ ] 已签收订单被正确拒绝并提示走售后
- [ ] 有效订单被正确取消并触发退款
- [ ] 退款信息（金额、到账时间）正确返回

### 6.5 意图识别验收标准
- [ ] 正确识别 logistics、urgent、cancel 三种意图
- [ ] 正确提取订单号实体
- [ ] 正确提取用户诉求细节
- [ ] 置信度低时返回澄清问题

### 6.6 数据层验收标准
- [ ] 数据接口抽象清晰，可通过配置切换模拟/真实数据源
- [ ] 模拟数据覆盖主要业务场景（不同订单状态、物流轨迹）
- [ ] 新增数据源时只需实现接口协议，无需修改业务逻辑
